1. Rebasing vs. MergingThis is the most common interview question.Merge: Creates a new "merge commit." It preserves the exact history of when branches joined but can make the commit graph look like "spaghetti" in large teams.Rebase (git rebase): Re-writes history by moving your commits on top of the latest commit of the target branch.Why SREs love it: It keeps a linear history, making it much easier to debug or revert changes if a deployment fails.The Golden Rule: Never rebase branches that have been pushed to a public/shared repository.2. Cherry-Pick: The "Surgical" Toolgit cherry-pick <commit-hash>The Use Case: You fixed a critical bug in a "Feature" branch, but that branch isn't ready for production yet. You use cherry-pick to grab only that specific fix and apply it to the main branch.Pro Tip: In DevOps, this is often used to backport security patches to older versions of an application.3. Git Stash: The "Pause" Buttongit stashThe Use Case: You’re halfway through a configuration change when a production "fire" happens. You need to switch branches immediately but aren't ready to commit your messy code.Advanced usage:git stash pop: Apply and remove the last stash.git stash list: See all hidden work.git stash apply: Apply the work but keep it in the stash (safer).4. Git Bisect: The Debugger's Secretgit bisectThe Scenario: "The app worked yesterday, but today it's crashing. We don't know which of the 50 commits broke it."The Workflow: Bisect performs a binary search through your commit history.git bisect startgit bisect bad (current version is broken)git bisect good <hash_from_yesterday>Git checks out a middle commit. You test it. You tell Git good or bad.Repeat until Git points to the exact "First Bad Commit."5. Git Hooks: The DevOps Entry PointHooks are scripts that run automatically during Git events. They are located in .git/hooks/.Pre-commit: Used to run Linters or Unit Tests. If the code is ugly or tests fail, the commit is blocked.Pre-push: Used to check for Secrets (like AWS keys) accidentally hardcoded in the code.Post-receive: Used on the server-side to trigger a CI/CD pipeline (Jenkins/GitHub Actions).6. Git Flow vs. Trunk-Based DevelopmentStrategyHow it WorksBest ForGit FlowUses feature, develop, release, and hotfix branches.Complex products with scheduled release cycles.Trunk-BasedDevelopers push small, frequent updates to a single main branch.SRE/DevOps favorites. Best for CI/CD and high-velocity teams. Requires heavy automated testing.7. Monorepo StrategiesA Monorepo is one repository containing multiple projects (e.g., Google or Meta).Pros: Easy to share code between services; atomic changes (change API and Frontend in one commit).Cons: Giant repo size; requires advanced tooling like Bazel or Nx to only build/test the parts of the code that actually changed.Placement "Killer" Question:Interviewer: "You accidentally committed a 500MB database file or an AWS Secret Key to Git. You deleted it in the next commit, but the repo size is still huge. How do you fix it?"The "15 LPA" Answer:"Simply deleting it in a new commit doesn't work because Git stores the history. I would use git filter-repo (or the older git filter-branch) to scrub the file from the entire commit history. After that, I’d force-push the clean history and rotate any compromised credentials immediately."
