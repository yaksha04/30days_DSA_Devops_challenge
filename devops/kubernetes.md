 1. Architecture: The Brain and the BrawnA Kubernetes cluster is divided into two main parts:The Control Plane (The Brain)This is the "management" layer that makes decisions about the cluster.API Server: The front door. Everything (users, nodes, internal components) talks to this.etcd: The cluster's database. It stores the "Source of Truth" for every configuration.Scheduler: The matchmaker. It decides which node a new Pod should live on based on available resources.Controller Manager: The watchman. It ensures the "current state" of the cluster matches your "desired state."Worker Nodes (The Brawn)These are the machines (VMs or physical servers) where your applications actually run.Kubelet: An agent that runs on each node to ensure containers are running in a Pod.Kube-proxy: Handles the networking rules so Pods can talk to each other and the outside world.Container Runtime: The engine (like containerd) that pulls images and starts containers.📦 2. Workload Objects: Pods & DeploymentsPodsThe Pod is the smallest unit in Kubernetes. It’s a wrapper around one or more containers (usually just one). Containers in the same Pod share the same IP address and storage volumes.Note: You almost never create a Pod directly in production. Why? Because if a solo Pod dies, it stays dead.DeploymentsA Deployment is a manager for your Pods. You tell the Deployment, "I want 3 copies of my web app," and it handles the rest.Self-healing: If a Pod crashes, the Deployment restarts it.Scaling: You can scale from 3 to 30 Pods with one command.Rollouts: It handles zero-downtime updates by swapping old Pods for new ones one by one.🌐 3. Networking: ServicesPods are ephemeral (they die and get replaced). When a Pod is replaced, its IP address changes. Services provide a permanent, stable IP address or DNS name to reach your Pods.ClusterIP: (Default) Internal access only.NodePort: Opens a specific port on every node's IP (accessible from outside).LoadBalancer: Provisions a real cloud load balancer (AWS/GCP/Azure) to route traffic to your app.🔐 4. Configuration: ConfigMaps & SecretsYou should never hardcode passwords or API URLs inside your container images. Kubernetes decouples "code" from "config":FeatureConfigMapSecretPurposeNon-sensitive data (URLs, port numbers).Sensitive data (Passwords, API keys, SSH keys).SecurityPlain text.Base64 encoded (and can be encrypted at rest).UsageInjected as Environment Variables or Files.Injected as Environment Variables or Files.🏢 5. Namespaces: Virtual ClustersNamespaces allow you to partition a single physical cluster into multiple virtual ones. This is great for:Environment Isolation: Keeping dev, staging, and prod separate on the same hardware.Team Isolation: Ensuring Team A doesn't accidentally delete Team B's apps.Resource Quotas: Limiting how much CPU/RAM a specific team can use.The Big Picture SummaryYou define a Deployment in a YAML file.The Control Plane schedules Pods onto Worker Nodes.A Service gives those Pods a stable entry point.ConfigMaps/Secrets feed the Pods their settings.Everything is organized inside a Namespace.
