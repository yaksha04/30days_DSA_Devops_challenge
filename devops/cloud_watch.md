1. CloudWatch Metrics, Logs, and Alarms
A. CloudWatch Metrics

Metrics are numerical data points collected over time to monitor AWS resources and applications.

Examples:

EC2 CPU Utilization
Network In/Out
Disk Read/Write Operations
Lambda Invocations
RDS CPU Usage

Example:

EC2 CPU

10 AM → 20%
11 AM → 40%
12 PM → 85%

CloudWatch stores these values as metrics and lets you graph them.

Types of Metrics
AWS-provided metrics (EC2, S3, Lambda, etc.)
Custom metrics (published by your application)

Example of a custom metric:

Number of active users
Orders processed per minute
B. CloudWatch Logs

Logs contain detailed text generated by applications and AWS services.

Examples:

Application Log

User logged in
Database connected
Payment failed

Sources include:

EC2
Lambda
ECS/EKS
API Gateway
VPC Flow Logs

Logs help troubleshoot issues after they occur.

C. CloudWatch Alarms

Alarms monitor metrics and trigger actions when thresholds are crossed.

Example:

CPU > 80%

↓

Alarm

↓

Send Email

Actions can include:

Send SNS notification
Stop/start/reboot EC2
Trigger Auto Scaling
Invoke a Lambda function

Example:

CPU > 80% for 5 minutes

↓

Send alert to DevOps team

B. CloudWatch Logs Insights

Logs Insights lets you query CloudWatch logs using a query language.

Example query:

fields @timestamp, @message
| filter status = "ERROR"
| sort @timestamp desc

This returns only error log entries.

Useful for:

Finding exceptions
Investigating failed requests
Counting occurrences
Performance analysis
3. AWS X-Ray (Distributed Tracing)

X-Ray traces requests across multiple services in a distributed application.

Example architecture:

Client

↓

API Gateway

↓

Lambda

↓

DynamoDB

↓

S3

Without X-Ray:

You know the request failed.
You don't know where.

With X-Ray:

API Gateway → 20 ms

Lambda → 50 ms

DynamoDB → 300 ms ❌

S3 → 10 ms

It identifies where latency or errors occur.

Features:

End-to-end tracing
Latency analysis
Dependency maps
Error visualization

Common interview question:

Why use X-Ray?

To identify bottlenecks and trace requests across distributed microservices.

7. SNS and SQS for Alerting
Amazon SNS (Simple Notification Service)

SNS is a push-based messaging service.

Example:

CloudWatch Alarm

↓

SNS

↓

Email

SMS

Lambda

When an alarm triggers, SNS immediately pushes notifications to subscribers.

Common uses:

Alert emails
SMS notifications
Trigger Lambda
Notify multiple systems simultaneously
Amazon SQS (Simple Queue Service)

SQS is a message queue that stores messages until they are processed.

Example:

Application

↓

SQS Queue

↓

Worker

↓

Database

Advantages:

Decouples applications
Buffers traffic spikes
Ensures reliable message delivery
SNS vs SQS
SNS	SQS
Push-based	Pull-based
Sends notifications immediately	Stores messages until consumed
One-to-many	One consumer per message (standard pattern)
Used for alerts	Used for asynchronous processing
Together

Interview Questions
1. What is the difference between CloudWatch Metrics and Logs?
Metrics are numerical time-series data (e.g., CPU utilization, request count).
Logs are detailed text records generated by applications and services for debugging and analysis.
2. What is a CloudWatch Alarm?

A CloudWatch Alarm watches a metric and performs an action (such as sending an SNS notification, invoking a Lambda function, or triggering Auto Scaling) when a defined threshold is reached.

3. Why is AWS X-Ray used?

AWS X-Ray traces requests across distributed applications, helping identify latency, bottlenecks, and errors in microservices.

4. What is the difference between CloudTrail and CloudWatch?
CloudTrail records who performed which AWS API action and when (auditing).
CloudWatch monitors resource performance, metrics, logs, and alarms (operations).
5. What is AWS Config used for?

AWS Config records resource configurations, tracks configuration changes over time, and evaluates resources against compliance rules.

6. What is the difference between AWS Health Dashboard and Trusted Advisor?
AWS Health Dashboard informs you about AWS service events and maintenance that affect your account.
Trusted Advisor provides recommendations to improve cost, security, performance, resilience, and service limits.
7. When would you use SNS versus SQS?
Use SNS when you need to immediately notify multiple subscribers (email, SMS, Lambda, HTTP endpoints).
Use SQS when you need reliable, asynchronous message processing and decoupling between applications.
